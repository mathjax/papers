STEM is not Niche!

  • STEM is a cornerstone of education
  • But also of information in general
    • Encyclopedic entries
    • Data visualisations in every day news (e.g., Covid)
    • Infographics, Interactive articles
  • 20% live with some disability
  • Overcoming the barriers that we build for any needs
    • Visual, Aural, Movement, Cognition
    • Temporary and Situational disabilities
  • Content is moving online, Covid-19 is a serious driver behind this

Why is STEM Hard?

  • STEM brings together some of the hardest a11y problems
    • Specialised vernacular, scripts, tables, formulas, diagrams
    • Animations, simulations, Interactive exercises
  • STEM are infinite languages
    • meaning shifts, vernacular is added
    • aimed at diverse audiences (pubils, students, scientists)
    • difficult to understand (in particular automatically)
    • precision is important (vague recognition is not enough)
  • Standard declarative approaches often fail
  • Even if WCAG is satisfied, STEM is rarely accessible
  • Many solutions are good for K-12 only

Historical Solutions

  • Learning and reading material in Braille or large print
  • Textbooks audio recorded by expert readers
  • Use of models, tactile prints, swell paper, etc. in class
  • Experts for writing subtitles
  • Complex content could only be interpreted/explained by a human
  • Exams: Specialist support needs to be provided, often one-on-one

All this is already a hindrance to inclusive education and worsens in modern education

The Present and Future

  • Anyone can prepare material, often on the fly
    • Static out: Textbooks, academic publishing, research papers, proceedings
    • Ephemeral in: lecture notes, handouts, forums, blogs, etc.
  • Moving to web based learning management systems
  • Students are expected to work through material independently
  • Even more barriers:
    • Aural: Subtitling your lecture video
    • Movement: Not everyone can use a mouse

The Web: Threads and Opportunities

Thread: Danger that the Web widens the accessibility gap

  • Little assistive technology support for specialist content
  • Needs to be usable by everyone and available instantly

Opportunity:

  • Web is ubiquitous, formats are rich
  • Allow for hidden information
  • Representations and display are flexible

Exploit the Structure of the Web

Crudely we can identify three "layers" on the web

  • Syntax
    • The basic content, i.e., text, images, icons, ...
  • Styling
    • How to render it, via CSS, (JavaScript), SVG
  • Semantics
    • Its meaning: Special tags, ARIA, Microdata, RFDa, data attributes, ...

Where does STEM fit?

  • Declarative syntax representations have to fail (negative example: MathML)
  • Let's embed information as semantics

Moving Content Online

  • Poor choices: Putting print formats online
  • Good choices: Transforming content to web content
  • Best choices: Authoring for the web

When settling for good:

  • Avoid just replicating print layout on the web
  • Transform content that allows for rich presentation.
  • The web is syntax, the semantic has to come from elsewhere
    • Note: LaTeX, MathML, CML, MOL, etc. are all only syntax!
  • Embed data that allows for middleware between browse and AT

Good Practice Aims

  • Convert content without loosing information
  • Embed suitable rich semantic that allows
    • Accessibility for users with diverse needs
    • Generation of alternative formats: tactile, audio-tactile, sub-titles etc.
    • Can be exploited by other software

Overarching Goals:

  • Source independence: Use what authors produce
  • Tool independence: No special tools to author or read in
  • Platform independence: Work on all OS, browsers, AT, ...

Let's Start with Maths

Initially concentrate on formulas:

  • Voicing can be difficult
  • Small expression can lead to long text
  • Interaction and exploration is important
  • Different "views" can be helpful:
    • Blind readers need more disambiguation than dyslexic users
  • Highlighting helps to separate symbols
  • Magnification of semantic components

  • Math markup (e.g., LaTeX, MathML) are not very expressive
  • SRE rewrites internally into semantically improved term structure
    ax^2+bx+c=0

  <mo>+</mo>

Render Accessibly

  • For example use MathJax
  • A JavaScript library for rendering Mathematics in all browsers
  • Input: LaTeX, AsciiMath, and MathML
  • Output: HTML/CSS, SVG
  • De facto rendering solution of (nearly) all Mathematics on the web
  • Uses Speech Rule Engine for semantic enrichment and speech

Other Solutions

  • Screen readers with math support
    • ChromeBooks: ChromeVox
    • Windows: Jaws, NVDA/MathPlayer, Narrator
    • Apple: VoiceOver
  • Specialist tools: EquatIO, InftyReader, etc.

Conversion Good Practice

  • Use a tool that retains sources and minimises clutter
  • Try to keep source even if you need to go to a different format
  • Postpone conversion as long as possible
  • Convert in bulk, not equation by equation
  • DO NOT be afraid of the command line!
  • Make sure the output format is accessible
  • Convert and test often
    • E.g., pandoc allows you to immediately work with MathJax
pandoc --standalone --mathjax -o example.html example.tex

Conversion Caveats

  • Avoid HTML export methods, e.g.,
    • LibreOffice's export to HTML
    • Google Doc's Download as zipped HTML
  • Stay away from online tools, even if they promise "Clean HTML"
  • Avoid tools that turn Maths into images, even with alt text
  • If possible, avoid plug-in tools like MathType. They lead to clutter!

Do some simple tests that consume very little time

  • Zoom
    • How does content react when zooming in or out?
    • Just a few key strokes
  • High Contrast
    • What if you change the colour palette?
    • One Chrome extension, one key stroke
  • Keyboard interaction
    • Can you reach everything without a mouse?
    • Multiple keystrokes

Offline, Epub, etc.

  • Epub3 is very similar to HTML5
  • Most conversion workflows also allow for Epub
  • However, you need to convert the Math yourself
  • E.g., use MathJax npm package server side

Best Practice: Epub

  • Convert to SVG (scalable vector graphics)
  • Try to generate "deep" speech
  • Again: The later you convert the Maths, the better the results

LMS

  • Most learning mangagement systems have provisions for Maths
  • Many use MathJax somewhere (Blackboard, Canvas, Moodle)
  • How content is presented depends on local setup
  • Some features might not be available
  • Talk to your technical staff and be sure what you need!

Best Practice: LMS

  • Continue to use your regular authoring workflow
  • Check for keyboard accessibility
    • In particular for quizzes and exams

What about Diagrams?

Diagrams: Caveats

  • Images with ALT-text satisfies WCAG but is generally not enough
  • Avoid Bitmaps

Diagrams: Better Practice

  • SVG is a suitable container format
    • Highly structured
    • Embedding of hierarchical content
    • Foreign objects embedded via namespaces
    • Muliple views on content to support diverse audiences
  • Check what your authoring software can export

SVG: Good Practice

  • Not every SVG is a good SVG
  • SVG can be flat and without information

Diagrams: Best Practice

  • SVG should be well structured
  • Structure should reflect the structure of data
  • Again: Do not throw away your intial source
  • Try to export as much as you can
  • Try to embed data or sources, e.g., as XML in alternative namespaces

Authoring for the Web: Semantics and structure

Master the basics, e.g., Principles of Accessible Design:

  • Provide appropriate alternative text
  • Provide appropriate document structure
  • Provide headers for data tables
  • Ensure links make sense out of context
  • Caption and/or provide transcripts for media
  • Do not rely on color alone to convey meaning
  • Make sure content is clearly written and easy to read

  • every writing tool offers some form of structuring but authors might not know or use it correctly
    • essential for a11y to understand what benefits
  • HTML5 structure is simple and robust (yet not everything is equally useful)
    • TOC, navigation
    • Sectioning/landmarks, headings - less so: asides, footer, addresses
    • Paragrpahs, lists, quotes, figures, tables, preformatted text
    • timed media
    • text: emphasis, abbreviation, definition - less so: citation, code, kbd, samp, var

Authoring for the Web: Maths

Authoring with the Web in mind

  • Avoid authoring for print only
  • Try to leave spacing to a layout engine
  • Never overlay symbols!
  • When using LaTeX: Avoid homegrown macros
  • Reduce use of specialist packages

Other Things to Avoid

  • Infinite Scrolling
  • Disabling Zoom
  • Referencing by colour name only
  • Elements that are visually first but syntactically last
  • Announcements on page load

Conclusions

  • Basic mechanisms for accessible STEM on the web are available
  • We just have to use them right
  • Let's not be too hung-up on standards
  • Think more about the information that you want to convey
  • Always avoid limiting the creativity of authors
  • We need more emphasis on
    • automatic interpretation,
    • machine learning and understanding
    • implementation of flexible workflows

Conclusions

Make it Future Proof:

  • Tools are constantly improving
  • Requirements are changing
  • The more information is in your sources, the better
  • Retain as much from your sources as possible
  • Export information rich formats from your authoring software
  • There might be use for it in the future or other applications
